# BioCSSwitch Skills Audit

## Scope and method

This audit covers everything under `packs/`: all 21 `SKILL.md` files (20 production skills across 19 packs, plus one infra smoke-test skill), all 19 `pack.json` manifests, the shared `_lib/` framework (12 modules), and the MCP server implementations behind each skill. Every tool name a `SKILL.md` instructs the model to call was cross-checked against the ~169 tools actually registered in source (`@server.tool(...)` decorators, including the factory-generated ones in `bio-gene/ncbi_server.py`). This is a static, source-level review — the MCP servers and the host loader that reads `pack.json` (referred to in-repo as "CSSwitch" / "Science") were not executed, so statements about install-time dependency resolution are inferred from manifest data, not observed runtime behavior.

## Executive summary

This is an unusually disciplined skill set for a biomedical research assistant. The organizing idea — every research conclusion must be traceable to a tool-verified citation, typed by evidence level from upstream metadata (not text heuristics), bounded by an explicit applicability boundary, and closed out with a mandatory uncertainty panel — is enforced fairly consistently across 20 skills and backed by real deterministic code (`_lib/extrapolation_checker.py`, `_lib/methodology_checker.py`, `_lib/critique_scoring.py`), not just prompt instructions. Several packs (`bio-ml`, `bio-scfm`, `bio-spatial`, `bio-sc-downstream`) have perfect 1:1 parity between what the skill teaches and what the server implements.

The problems are concentrated in three places: (1) a handful of concrete code/doc mismatches that will produce wrong or degraded behavior in specific skills, (2) a consistent pattern of well-built tools that no skill ever tells the model to call, and (3) thin test coverage and uneven documentation depth relative to the size of the codebase (~18,400 lines of Python). None of this undermines the architecture; it's the kind of drift you'd expect in a fast-moving 19-pack system, and all of it is fixable without redesign.

## What's working well

The anti-hallucination posture is real, not just stated. `evidence_verify` treats `exists=false` as final and explicitly refuses to suggest "similar" IDs (`bio-audit/evidence_verify_server.py`); evidence typing comes strictly from PubMed MeSH `publication_type` or CT.gov `study_type`, never from title keywords; `grade_server.py` structurally blocks upgrading RCT-derived certainty (`"研究设计属 RCT 体系，GRADE 规定不可升级——已忽略 upgrades"`), matching the rule stated in `grade-sof/SKILL.md`. Generated analysis skeletons (ML, single-cell, spatial, scFM) embed a literal `raise SystemExit("SKELETON...")` guard so the model cannot claim a script ran when it hasn't — a small mechanism that does a lot of work against a very common failure mode (LLMs narrating fabricated results). The fixture-replay layer (`_lib/fixtures.py`) redacts API keys before recording, enabling offline, deterministic tests without leaking secrets. The `uncertainty_ledger` tool auto-derives most of the five-part panel from `evidence_graph` output rather than leaving it to the model's memory, which is exactly the right place to put that logic. Provenance hashing (`_lib/provenance.py`) is genuinely careful about reproducibility (canonical JSON, sha256, an honest distinction between a metadata "descriptor fingerprint" and a real content hash). Coverage breadth is also worth noting on its own: 19 packs span literature, trials, drug/target/gene/protein databases, ontology normalization, PHI handling, GRADE, critique, four single-cell sub-domains, ML strategy, and a causal knowledge graph — a coherent stack, not a grab-bag.

## Confirmed defects

### 1. `uncertainty_ledger`'s `language` parameter is dead — every workflow closes in Chinese regardless of skill language

`uncertainty_ledger` (`bio-audit/evidence_verify_server.py`, ~L684–768) accepts `language: "zh"|"en"` in its input schema, but the function body never branches on it: the Markdown section titles come from a hardcoded `titles_zh` dict, and most generated item strings (`"—— 证据等级"`, `"无有效引用支持，尚不能确证"`, etc.) are hardcoded Chinese. Compare this to `evidence_build_table` in the same file, which correctly branches `headers_zh`/`headers_en` on the `language` argument — so the bilingual pattern exists in the codebase, it just wasn't applied here. Every one of the 8 `bio-workflows` skills, plus `lit-review`/`target-discovery`/etc., mandates closing with this exact panel (`"结论必须落成..."` / "缺这五段，回答视为未完成"). Practically: an English-speaking user running the fully-English `scientific-critique` or `spatial-analysis` skill will get an English answer that ends in a Chinese-titled panel, because the tool that renders the mandatory closing section ignores the language it was asked for. Fix is localized — thread `language` through the title dict and the ~10 hardcoded strings in the render block.

### 2. `trial-landscape` tells the model a capability doesn't exist, and prescribes a worse workaround for something that already works

`bio-workflows/skills/trial-landscape/SKILL.md` (Step 4) says endpoint aggregation "isn't directly exposed in CSSwitch," and instructs the model to fetch every matching NCT individually via `ctgov_detail` and hand-aggregate `primary_outcomes`. That claim is false: `bio-mcp-shim/ctgov_shim.py` registers a tool literally named `analyze_endpoints` that does exactly this — one call, server-side `Counter` aggregation of primary/secondary outcome measures across all matching trials. The root cause is architectural: ClinicalTrials.gov access is implemented twice under two different tool-name surfaces that were clearly built at different times — `bio-trials` (`ctgov_search` / `ctgov_detail` / `ctgov_by_sponsor`, home-grown naming) and `bio-mcp-shim/ctgov_shim.py` (`search_trials` / `get_trial_details` / `search_by_sponsor` / `search_investigators` / `analyze_endpoints` / `search_by_eligibility`, deliberately named to impersonate Anthropic's hosted remote MCP). `trial-landscape` only knows about the first surface, so it recommends an N+1-call workaround for a feature the second surface already solved in one call. This isn't just inefficient — it's also the pack most likely to hit CT.gov rate limits given `trial-landscape`'s own KOL-discovery step already issues one `ctgov_detail` call per trial.

The deeper issue worth flagging is the duplication itself, not only the stale doc. `bio-mcp-shim` exists specifically to impersonate Anthropic's remote MCP servers under virtual-login constraints, so it may not always be installed alongside `bio-workflows` — meaning the durable fix probably isn't "just reference `analyze_endpoints`," it's deciding whether endpoint-aggregation belongs in the foundational `bio-trials` pack (available whenever trial search is available at all) rather than only in the shim layer.

### 3. `bio-workflows/pack.json` under-declares dependencies that its own skills require

`bio-workflows`'s manifest lists `dependencies: ["bio-lit","bio-audit","bio-singlecell","bio-sc-downstream","bio-scfm","bio-sc-atlas","bio-spatial"]` and ships zero servers of its own (`"servers": []` — it's a pure skill bundle). But `target-discovery`, `trial-landscape`, and `grant-specific-aims` explicitly instruct calling `ols_search`/`hgnc_lookup_symbol`/`disambiguate` (`bio-norm`), `chembl_*`/`ot_*`/`rxnorm_*`/`fda_label` (`bio-drug`), `uniprot_entry` (`bio-gene`), and `ctgov_*` (`bio-trials`) — none of which are declared dependencies. `geo-triage` similarly calls `geo_search`/`geo_summary` (`bio-gene`, undeclared). All of the tool *names* referenced are real (verified against source — this is not a hallucinated-API problem), so if the host resolves dependencies loosely this may be invisible in practice. But as written, the manifest is not an accurate map of what four of the pack's eight skills actually need, which will bite the moment dependency resolution is tightened or used for install-time capability checks.

### 4. Well-built tools that no skill ever mentions

This shows up in at least five places, and it's a pattern, not an isolated miss:

- `design_counter_experiment` and `critique_checklist` (`bio-critique/critique_server.py`) are fully implemented — the former sits on top of a dedicated, well-designed module (`_lib/counter_experiment.py`) that maps each extrapolation rule to a concrete falsification-study design (design type, endpoint, sample-size formula, a live ClinicalTrials.gov search link). `scientific-critique/SKILL.md`'s workflow never mentions either tool, only the other 7 of the pack's 9 tools. Counter-experiment design is arguably the single most actionable output `bio-critique` can produce, and it's currently undiscoverable from the skill text.
- `grade_evidence_dossier` and `etd_probabilistic_recommendation` (`bio-audit/grade_server.py`) — the latter accepts full probability distributions over certainty for a proper probabilistic Evidence-to-Decision recommendation — are absent from `grade-sof/SKILL.md`, which only walks through `grade_outcome`/`grade_explain`/`grade_sof_table`/`etd_recommendation`.
- `sc_workflow_recipe` (`bio-singlecell/singlecell_server.py`), which generates a runnable Snakemake/Nextflow package rather than a loose script, is absent from `single-cell-prep/SKILL.md` entirely.
- `evidence_linter.py` is a genuinely well-scoped standalone CLI ("scan draft.md / rebuttal.md / grant aims for fake PMIDs before submission," with `--strict` for CI-style nonzero exit) whose own docstring names exactly the use cases `grant-specific-aims` and `reviewer-response` cover — pre-submission and pre-resubmission citation sweeps. Neither skill mentions it, even though `evidence-audit` itself says fabricating a grant PMID is worse than a rebuttal one.
- `kg_query` (`bio-kg/knowledge_graph_server.py`) is the one read-only tool in the KG pack that `causal-knowledge-graph/SKILL.md` never surfaces, despite the workflow implying the model should be able to look up what's already stored before extracting new triples.

None of these are bugs in the tools themselves — they're finished features with no path for the model to discover them from the skill text it's actually given.

### 5. "Multi-agent debate arena" is referenced twice, implemented nowhere

`experimental-design/SKILL.md` step 2 says: "If the claim is contested, use `bio-critique` or the multi-agent debate arena before selecting the hypothesis." `experiment_server.py`'s own `agentic_experiment_plan` output repeats the phrase in its returned roadmap text (`"multi-agent debate can propose rival hypotheses"`, L662). No tool anywhere in the ~169 registered across all packs has "debate" in its name or description. Either this names a real planned feature that hasn't shipped, or it's meant as a reasoning framing the model should perform itself rather than a callable capability — but as written it reads like an instruction to use a system feature that doesn't exist, which is exactly the kind of phrasing this project is otherwise careful to avoid (compare `target-discovery`'s explicit "CSSwitch 目前**没有**" callouts when a capability is genuinely absent).

### 6. `sensitive-mode`'s audit trail hashes the input but not the free-text fields next to it

`audit_log_write` (`bio-privacy/audit_server.py`) SHA-256's and discards `input_sample` exactly as `sensitive-mode/SKILL.md` promises. But `summary` and `extra` — the fields the *model* fills in — are stored in plaintext (truncated at 200 chars, not scanned). The skill's own "坑" section acknowledges this is a social contract, not a technical one ("工具会截断超长 extra 值，但语义合规靠你自己" — the tool truncates long values, but semantic compliance is on you). For a privacy-critical pack, that's a thin guarantee: nothing stops a `summary` string from containing the patient name the whole flow exists to protect. This is enforceable in code — `audit_log_write` could run its own internal `phi_scan()` over `summary`/`extra` string values before writing and redact or reject high-confidence hits, turning the current documentation-only safeguard into defense in depth.

### 7. Test coverage is thin relative to how testable this codebase is

`packs/` contains exactly two unit test files (`_lib/test_extrapolation_checker.py`, `_lib/test_methodology_checker.py`) for roughly 18,400 lines of Python. The project's own stated design philosophy — deterministic, stdlib-only, pure functions wherever possible — is about the best-case scenario for unit testing, and most of it has none: `_lib/critique_scoring.py`, `_lib/causal_kg.py`, `_lib/evidence_profile.py`, and `bio-compiler/compiler_lexicon.py`'s entity/archetype detection are all pure, deterministic, and currently unverified by any test in this tree. Separately, `bio-workflows/generators/README.md` references a golden-fixture test suite at `test/generators/test_generators_golden.py` that isn't inside `packs/` at all — worth confirming that suite still exists and runs in CI, since it isn't visible from (or shipped with) this folder.

### 8. Minor code-quality nit

`bio-compiler/compiler_lexicon.py`, `detect_genes()`: the `elif len(tok) >= 3 and any(ch.isdigit() for ch in tok) or tok in KNOWN_TARGETS:` clause has unreachable logic — any token in `KNOWN_TARGETS` was already handled by the preceding `if`, so `or tok in KNOWN_TARGETS` in the `elif` can never fire. Harmless (the `and`/`or` precedence doesn't change behavior here), but worth cleaning up since it reads as intentional and isn't.

## Aspects that can be compensated for later

These aren't bugs — they're places where the current design is a reasonable v1 that has visible room to grow.

**Skill depth is uneven, and not strictly along the Chinese/English split it looks like at first.** 14 of 20 production skills are written in Chinese, 5 in English (`scientific-critique`, `experimental-design`, `causal-knowledge-graph`, `biomedical-ml-strategy`, `spatial-analysis`), 1 is language-agnostic infra. The Chinese skills — especially `evidence-audit`, `grade-sof`, `question-compiler`, `trial-landscape`, `grant-specific-aims` — share a distinctive, effective template: numbered "iron rules," a worked good example, a worked bad example, and an explicit boundary/edge-case section. `biomedical-ml-strategy` and `spatial-analysis` (English) hit the same bar with long Do/Don't lists and named failure modes. But `scientific-critique`, `experimental-design`, and `causal-knowledge-graph` are noticeably thinner — workflow steps and a rule list, no worked examples, no boundary section. Since the template clearly exists and works elsewhere in the same repo, bringing these three up to it is low-risk, high-consistency work. Separately, it's worth deciding explicitly whether skill language should track the end user's language (in which case the `uncertainty_ledger` bug above needs fixing first) or whether Chinese is intentionally the working language regardless of user — right now it reads like an artifact of when each skill was written rather than a deliberate choice.

**`compiler_lexicon.py`'s entity lexicons are small and oncology/neurology-heavy.** `DISEASE_ABBR` has 24 entries, `DISEASE_ZH` 18, `KNOWN_TARGETS` ~65 genes, `KNOWN_DRUGS` ~23, and the 7 `ARCHETYPES` categories have no bucket for infectious disease, psychiatry, surgery/devices, or nutrition — those questions will fall through to `"unknown"` and get a generic "search PubMed" routing. The skill already discloses this honestly ("覆盖不到的实体会标 candidate"), so this isn't a false claim, just a coverage ceiling. Given `bio-norm` already wraps live OLS/HGNC/MeSH lookups, a natural next step is using those as a fallback for entities the static lexicon misses, rather than only widening the hardcoded lists.

**The mandatory five-part uncertainty panel doesn't obviously fit every workflow it's bolted onto.** It's clearly right for evidence-generating workflows (`lit-review`, `target-discovery`, `trial-landscape`, `geo-triage`, `sc-analysis`). It's a less natural fit for `reviewer-response`, which produces a point-by-point rebuttal letter, not a fresh research conclusion — appending "Known knowns / Known unknowns / Conflicts / Missing data / Next experiment" after a rebuttal risks looking bolted-on rather than useful. `grant-specific-aims` has a milder version of the same tension: the Aims page itself already *is* the "next experiment" proposal, so a trailing "Next experiment" bucket can end up restating the Aims. Worth considering whether the mandate should be conditioned on "this response asserts a new scientific conclusion" rather than applied to all 8 `bio-workflows` skills unconditionally.

**The `bio-trials` / `bio-mcp-shim` split (see Defect 2) is worth resolving as a design decision, not just a doc fix.** Two independent ClinicalTrials.gov clients with different tool names and different capability sets is the kind of thing that tends to drift further apart, not converge, if left alone.

**`_lib/smoke/canary_skill/SKILL.md` is a phase-1 dev probe living inside what looks like the production skills tree.** It's well-guarded (fires only on an exact verbatim phrase), but it's explicitly dated ("用于 phase-1 路径验证") and framed as disposable. Worth confirming it's excluded from whatever build step ships packs to end users, rather than relying on the trigger phrase alone to keep it inert — and removing it once phase-1 verification is actually done.

**Cross-pack handoffs are directionally documented but not mechanically verified.** Skills reference each other by name constantly ("交给 `single-cell-prep`", "hand high-priority gaps to `bio-experiment.agentic_experiment_plan`"), which is good for a human reading the skill, but nothing checks that a referenced skill/pack is actually present at handoff time the way `evidence-audit` explicitly checks for `evidence_verify`'s presence and prints a warning banner if it's missing. Only `evidence-audit` and `question-compiler` have that "if the tool isn't connected, do X instead" fallback; the other 18 skills assume their dependencies are simply there.

## Suggested priority order

Highest leverage first: fix the `uncertainty_ledger` language bug and the `trial-landscape`/`analyze_endpoints` mismatch (both are small diffs with outsized, cross-skill visibility); correct `bio-workflows/pack.json`'s dependency list; then do a pass wiring the five orphaned tools into their skills' workflow sections (text-only changes, no code). After that, the example/depth pass on the three thin English skills and the `sensitive-mode` defense-in-depth change are the best next investments. Test coverage and lexicon expansion are real but lower-urgency — they compound over time rather than causing a specific wrong answer today.
